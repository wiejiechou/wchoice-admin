# 功能地圖 (v0.4)

## 選單結構圖 (v0.4)

```text
WChoice Admin
├── 報價項目管理 [ItemPortfolio] (/item-portfolio/index)
│   ├── 作業區域維護 [SpaceMaintenance] (/item-portfolio/space/list)
│   └── 服務項目維護 [ServiceItemMaintenance] (/item-portfolio/service-item/list)
├── 報價管理 [QuotationManagement] (/quotation/index)
│   └── 報價清單維護 [QuotationMaintenance] (/quotation/list)
├── 服務管理 [ServiceWorkspace] (/service-workspace/index)
│   ├── 服務單維護 [ServiceOrderMaintenance] (/service-workspace/order/list)
│   ├── 派工單維護 [DispatchMaintenance] (/service-workspace/dispatch/list)
│   └── 結案單維護 [CompletionMaintenance] (/service-workspace/completion/list)
└── 系統管理 [SystemManagement] (/system/index)
    ├── 使用者維護 [UserMaintenance] (/system/user/list)
    └── 角色維護 [RoleMaintenance] (/system/role/list)
```

- **命名慣例**: `[ComponentName]` 為路由名稱；`(/path)` 為瀏覽器訪問路徑與 Rest Path。

## 系統架構決策紀錄 (ADR)

- **命名原則**：採納 **L1 管理 / L2 維護** 之語意層次，區分業務模組與實體操作。
- **目錄結構策略**：採用 **層次化 (L1/L2) 佈局**。例如 `src/views-custom/item-portfolio/space/list.vue`。
  - _決策理由_：提升物理結構的「抗崩塌能力」，確保 Rest Path 與資料夾路徑 1:1 映射，利於權限攔截器之配置。
- **路由路徑慣例**：全面補完 L1 前綴（Prefix），確保 `/system/**/*` 等路徑匹配策略之嚴謹性。
- **操作邏輯**：L2 維護項預設入口均為 **GridView (list)**，新增/編輯採用 ReDrawer 等組件化操作。

# L1. 報價項目管理 (Quotation Items Management)

## 1.1 業務規劃

- **核心邏輯**：定義報價單所需的「底層組件」，實現「一次設定、可重複使用」的彈性。
- **維護對象**：
  - **作業區域維護**：定義施工的物理空間（如：美容室、會客區、玻璃區）。
  - **服務項目維護**：定義具體的服務內容、單位（座/面/坪）、單價與標準處理時間。
- **關聯機制**：一個服務項目可對應多個「條件備註」，這些備註將在報價單建立時成為微調參數（如：尺寸大、污漬重）。

## 1.2 技術實作

- **資料結構設計**：
  - `spaces` (作業區域): 只有基本欄位 `name` 與 `active_status`。
  - `service_items` (服務項目): 包含 `space_id` (FK), `unit`, `price`, `standard_time`。
  - `item_conditions` (項目備註/條件): 與 `service_items` 為一對多關聯。
- **UI 實作規範**：
  - 進入功能後先呈現 GridView (清單庫)。
  - 使用 ReDrawer 進行新增與編輯，減少頁面跳轉。

## 1.3 等待確認事項

- **套餐方案 (Packages)**：是否存在固定的「主打方案」(如：黃金鐵三角)？若有，需確認是否獨立出「套餐維護」功能，透過組件選擇快速帶入項目。
- **單價歷史**：當服務項目單價更新時，舊有的報價單是否需要鎖定歷史單價？(技術建議：報價單轉換為服務單時需 Snapshot 單價)。

# L2. 報價管理 (Quotation Management)

## 2.1 業務規劃

- **核心邏輯**：系統的核心故事出發點，管理從勘估到合約成立的過程。
- **維護對象**：
  - **報價清單維護**：綜合搜尋、狀態追蹤 (新建/報價中/報價成立/不成立)。
  - **報價單建立/編輯**：組合「作業區域」與「服務項目」，透過勾選備註條件調整最終報價。
- **關鍵流程**：
  - **現場照片上傳**：需支援多圖上傳，並指定一張為 PDF 封面的代表圖。
  - **輸出 PDF**：提供正式報價單下載供客戶簽簽核。

## 2.2 技術實作

- **狀態機設計**：嚴格控管報價單狀態流轉，當狀態變更為「報價成立」時，觸發 Trigger 產生「服務單」。
- **解決非同步覆寫**：針對照片批次上傳，採用「**本地累加器模式 (Local Accumulator Pattern)**」，確保 `props` 同步與狀態完整。
- **PDF 模組**：獨立入口於 GridView 操作欄，後端生成 PDF 以維持版面一致性。

## 2.3 等待確認事項

- **工時換算邏輯**：目前服務項目中有「處理時間」，但技術上無法區分「單人 vs 雙人」的換算倍率。業務上如何定義此係數？
- **電子簽章方案**：是否需要直接在系統中內建 Canvas 簽名功能，或是僅上傳簽回的照片/文件？

# L3. 服務管理 (Service Order Management)

## 核心設計意圖：為何是「服務 (Service)」而非「專案 (Project)」？

在架構設計的選型上，我們刻意選擇「服務」作為核心名詞，背後承載了對業務深層的理解：

- **業務的「重複性」與「週期性」**：維護產業並非一錘子買賣。同一間「店」會因為時間推演產生循環需求。使用 `Service` 能精準對接未來「定期保養」或「續約提醒」的週期性邏輯。
- **以「店」為中心的服務履歷**：我們的資料策略是**「以店為經、以次數為緯」**。透過服務單累積，我們能產出該場所的完整「醫學病歷」。開發者應以此同理心出發，理解我們是在記錄一間店的生命週期，而非孤立的任務。
- **未來的「合約化 (SLA)」擴展**：服務管理語境更易於與未來可能出現的「年約管理」或「服務等級協議」掛鉤，為系統預留了從「單次報價」進化到「長期訂閱」的架構空間。

## 3.1 業務規劃

- **核心邏輯**：控管報價成立後的執行品質與排程，是「派工量化」的關鍵。
- **維護對象**：
  - **服務單維護**：管理執行主體（以「店/場所」為對象），對應一對多報價/服務歷史。
  - **派工單維護**：媒合施工人員與施作檔期，產出施作契約。
  - **結案單維護**：完工後的滿意度調查收錄與歸檔。

## 3.2 技術實作

- **資料搬移策略**：報價單成立時，拷貝一份數據進入 `service_orders` 體系，實行「報價」與「執行」的數據區隔。
- **回填機制**：結案單提供 QR Code 提供客戶填錄 Google 表單，需思考是否透透過 API 或手動回填數據至系統進行分析。

## 3.3 等待確認事項

- **派工連動性**：派工單產生後，是否需要自動連動產生施工人員專用的「施工內容清單 (Checklist)」？
- **資料維度**：業主未來是否需要以「店」為維度檢視年度服務成本？這會影響資料庫在服務單層級的設計。

# L1. 服務單管理

## 描述

- 業務邏輯
- 執行管理的 `抽象單位` (名詞歸類陷入的抉擇，概念上 `專案 (project)`, `服務 (service)` 都符合 `管控的單位` )
- 當 `報價單` 的狀態變更為 `報價成立`，轉入 `服務追蹤維護`
  - 搬移一份資料 (原始欄位以 `報價單` 為主體)進入 `服務單`
  - 資料庫的設計上，會將 `報價` 與 `服務` 進行資料區隔 # 按經驗，會使用 `INSERT INTO A SELECT FROM B WHERE ID = ?`
- 觀察 `報價單` 的欄位設計，服務的對象主體是 `店 (營業場所)`
  - 一個 `客戶 (負責人)` 可以有 多間店
  - 一個 `店` 可以對應到多張 `服務單`
  - 從這樣的結構開展，要思考之後 `管理報表` 會用什麼樣的 `維度` 去調閱資料
  - 這部份成為了目前設計上的 `盲區` (即便業主沒有提出，我們仍需要有自己的一套 `資料管理策略`)

## 功能結構

- L2. 服務單清單 (GridView)

- L2. 派工單維護
  - 設定施工人員

- L2. 施工單列印
  - 尚無法確認，施工單的內容是否需要回填及列入維護內容

- L2. 結案單維護
  - 客戶可以透過 QRCode (Google 線上表單)，回填 `滿意度`

## 等待確認事項

- `派工單` 與 `施工單` 的產生時間是否具有連動性
- 關於 `服務單管理` 這個功能大項，目前只有取得 `產出後的結果`，所以在業務邏輯的故事上，偏向 `倒果為因`，以下為我們的理解
- 當 `報價單成立轉換為服務單`之後, 業務人員會採取以下步驟
  - 聯絡 `施工人員`, 確認可配合的人員及可以施作的檔期 (時間)
  - 確認施工人員之後，產生 `派工單` # 概念上類似給予施工人員的施作契約
  - 當施工人員要到案場的時候，產生 `施工單`，內容記載的資訊與 `報價單` 的項目相符，供施工人員做為執行業務的參考依據 (類似 check list)
  - `施工單` 上有部分的欄位，判斷應該是讓現場人員填寫，完成後，作為結案的歸檔紀錄
- 當 `施工完成`，進入 `結案單` 的流程
- 以上理解，可以作為第一版設計的依據
